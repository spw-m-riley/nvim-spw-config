make the setup easier to learn and maintain.

cheatsheet when you just need a quick nudge. The native helpfile is the

OVERVIEW                                    *onebeer-overview*

OneBeer is a full-featured Neovim setup built on |lazy.nvim|. Most config code
lives under the `onebeer` namespace in `lua/onebeer/`, while per-server LSP
settings live in `lsp/*.lua`.

The config is intentionally opinionated so it stays easier to extend without
guessing where things belong:

- one plugin spec per file under `lua/onebeer/plugins/`
- server-specific LSP settings under `lsp/*.lua`
- runtime helpers under `onebeer.utils`
- cross-cutting editor behavior in `onebeer.autocmds`
- machine-local changes in `lua/onebeer/local.lua`

The two built-in help surfaces give you a quick path and a deep path:

- |OneBeerHelp|   quick floating cheatsheet
- |onebeer|       full persistent manual

If you only remember one entrypoint, start here:

>
:h onebeer
<

STARTUP AND OWNERSHIP                       *onebeer-startup*
*onebeer-conventions*

`init.lua` is the entrypoint. It enables the Lua loader, then brings the
config up in a deliberate order so each layer arrives with what it expects:

1. `onebeer.utils`
2. `onebeer.settings`
3. diagnostics defaults
4. LSP defaults
5. `onebeer.config`
6. `onebeer.lazy`
7. `onebeer.health`
8. `onebeer.autocmds`
9. `onebeer.tools.commands`
10. optional `onebeer.local`

That load order also doubles as an ownership map:

- `lua/onebeer/settings/` owns editor defaults, theme-related settings,
  diagnostics defaults, and global LSP behavior.
- `lua/onebeer/plugins/` owns third-party plugin specs and their keymaps.
- `lsp/*.lua` owns server-specific configuration.
- `lua/onebeer/autocmds/init.lua` owns cross-cutting runtime behavior such as
  LSP attach mappings, write hooks, and statuscolumn handling.
- `lua/onebeer/tools/` owns config-local commands and generators.
- `lua/onebeer/local.lua` is reserved for machine-specific, non-tracked edits.

A few maintainer conventions help the repo stay calm over time:

- Put new plugins in their own file under `lua/onebeer/plugins/`.
- Use `onebeer.utils.safe_require()` instead of unchecked `require()` calls
  when a missing dependency should degrade gracefully.
- Use `onebeer.utils.map()` or the autocmd helpers instead of re-creating the
  same `vim.keymap.set()` / `nvim_create_autocmd()` boilerplate.
- Keep config-local helpers shared; do not duplicate shell wrappers or mapping
  helpers across plugin specs.

INSTALLATION AND FIRST RUN                  *onebeer-install*

Core requirements:

- Neovim 0.11.0 or newer
- `git`, `rg`, `fzf`, `gh`
- a Nerd Font

Recommended tools for a smoother first run:

- `mise` for Node.js and Go toolchains
- `brew` for system packages

First-run behavior you can expect:

- `lazy.nvim` bootstraps itself if not already installed.
- Mason installs the curated language-server set.
- `ts-install.nvim` auto-installs the configured Treesitter parsers.
- |:checkhealth| `onebeer` reports missing dependencies and can offer guided
  installs for supported tools.

Basic setup flow:

>
git clone <your-repo-url> ~/.config/nvim
nvim
:checkhealth onebeer
<

If you want GitHub-native features such as Octo or the GitHub Actions
language server integration, make sure this succeeds first:

>
:!gh auth status
<

After checkout, the help reference is ready right away via |onebeer|.

EDITOR DEFAULTS AND UI                      *onebeer-ui*
*onebeer-theme*
*onebeer-dashboard*
*onebeer-notifications*

Core editor defaults at a glance:

- leader is `<Space>`
- clipboard uses `unnamedplus`
- line numbers are enabled with relative numbers
- Treesitter-driven folding is enabled by default
- backups and persistent undo are enabled under `stdpath('data')`
- horizontal splits open below and vertical splits open right
- smooth scrolling is enabled
- a custom winbar is rendered by `onebeer.ui.winbar()`

Theme and chrome:

- Catppuccin Mocha is the default colorscheme.
- background transparency is enabled in the theme config.
- Blink/Copilot ghost text gets custom highlight groups.
- Lualine shows mode, filename, branch, diff, and attached LSP clients.
- the winbar shows file icon, path, modified marker, and readonly marker.

Dashboard and UI helpers that round things out:

- `snacks.nvim` owns the startup dashboard, notifier, scope highlighting,
  reference jumping, statuscolumn, and smooth-scroll controls.
- `mini.clue` shows a rounded key-hint popup for leader, `g`, `z`, marks,
  registers, and window commands.
- `tiny-inline-diagnostic.nvim` is enabled by default and replaces native
  virtual text until toggled off.

Useful UI entrypoints to keep handy:

Key		Meaning ~
`<leader>sd`	open Snacks dashboard
`<leader>un`	show notification history
`<leader>uv`	toggle native diagnostics virtual text
`<leader>uw`	toggle Snacks words/reference overlay
`<leader>us`	toggle statuscolumn visibility
`<leader>uS`	toggle Snacks smooth scroll
`<leader>cd`	toggle inline diagnostics
`<leader>uu`	toggle Undotree

A few behavior notes:

- Inline diagnostics start enabled and native virtual text starts disabled.
- When inline diagnostics are disabled, `CursorHold` falls back to a floating
  diagnostic window at the cursor.
- Insert mode temporarily hides the statuscolumn unless it has been manually
  disabled by the user toggle.

- `.envrc` -> `sh`
- `.log` -> `log`

- rounded borders are forced for LSP floating previews
- semantic tokens are enabled on attach when the server supports them
- inlay hints are enabled on attach when supported and can be toggled later
- codelens is enabled and refreshed on enter/hold/insert-leave when supported
- document highlights are enabled when the server supports them

Language-specific notes worth knowing:

Go                                          *onebeer-lsp-go*
- `gopls` enables aggressive inlay hints.
- analyses include `unusedvariable`, `unusedparams`, and `shadow`.
- staticcheck is enabled.
- codelenses include generate, gc details, govulncheck, tidy, vendor, and
  dependency upgrade helpers.

TypeScript / JavaScript                     *onebeer-lsp-typescript*
- `ts_ls` is the active TypeScript/JavaScript server.
- inlay hints are enabled broadly for parameters, variables, returns,
  properties, and enum members.
- rename/import/file-operation preferences are explicitly tuned for editor use.
- `typescript-tools.nvim` exists in the repo but is disabled.

Python
- `pyright` and `ruff` intentionally attach together for Python buffers.
- `pyright` keeps type analysis and `LspPyrightSetPythonPath`.
- `ruff` owns import organization and formatting handoff, while hover stays with
  `pyright`.

Rust
- `rust_analyzer` runs with build scripts and proc macros enabled.
- diagnostics prefer `clippy`.
- formatting stays disabled in the server because Conform owns `rustfmt`.

Ruby
- `ruby_lsp` stays tied to the active Ruby runtime.
- shared policy does not Mason-install a separate Ruby LSP gem runtime.
- formatting and on-type formatting stay disabled because Conform owns
  `rubocop`.

Zig
- `zls` is enabled when the executable is available.
- a local `zls.json` is picked up automatically when present at the workspace
  root.
- formatting stays with Conform / `zig fmt`.

Gleam
- Gleam is explicit partial support in wave 1.
- it keeps runtime-managed LSP + formatter + parser coverage.
- it does not add a separate `nvim-lint` layer.

SQL
- SQL is explicit partial support in wave 1.
- Treesitter + Conform (`sqlfluff`) are enabled.
- no SQL LSP is enabled in this wave.

Lua                                         *onebeer-lsp-lua*
- `lua_ls` treats the config directory and the Luv library as its workspace.
- third-party prompts are disabled.
- workspace scanning is intentionally limited for performance.
- formatting is disabled because |FormatToggle| uses Conform / Stylua instead.

GitHub Actions                              *onebeer-lsp-actions*
- Actions workflow YAML gets a dedicated `yaml.ghactions` filetype.
- the config resolves a GitHub token via `gh auth token` when available.
- the server inspects the repo remote to build repo metadata for workflow lint
  and language features.
- workflow support depends on the Actions language server executable chain and
  a working GitHub CLI login.

Treesitter and parser notes:

- `nvim-treesitter` is on the `main` branch and updates via `:TSUpdate`.
- `ts-install.nvim` auto-installs parsers for the configured language list,
  including Python, Gleam, SQL, Ruby, Rust, and Zig.
- `terraform-vars` is registered as the `terraform` language.
- `nvim-ts-autotag` auto-closes and renames matching tags in markup-like files.

AI, COMPLETION, AND SNIPPETS                *onebeer-ai*
*onebeer-completion*
*onebeer-snippets*

AI feature gating keeps assistant features available where they help and
quiet where they should stay quiet:

- `onebeer.config` computes whether Copilot is enabled.
- Copilot and Sidekick are disabled automatically when the current working
  directory is equal to or nested under `nopilot_dir` / `NOPILOT_DIR`.
- Tabnine support exists as a config flag but is not an active workflow.

Copilot and Sidekick entrypoints:

Key		Meaning ~
`<leader>cp`	open Copilot panel
`<leader>aa`	toggle Sidekick Copilot CLI
`<leader>ap`	open prompt picker
`<leader>ae`	explain code
`<leader>af`	fix diagnostics
`<leader>at`	write tests
`<leader>ar`	review current file or selection
`<leader>ao`	refactor code
`<leader>ac`	generate commit message

The prompt set covers explain, explain_error, fix, refactor, optimize,
tests, docs, types, review, review_selection, and commit-oriented prompts.

Completion behavior:

- `blink.cmp` is the active completion engine.
- LSP capabilities are extended through Blink.
- Copilot is added as a completion source when enabled.
- Lua buffers get `lazydev` support for Neovim/Lua development.
- documentation popups auto-show with rounded borders.
- completion is disabled in `minifiles` buffers.
- `<Tab>` first tries snippet jump, then Sidekick inline apply/jump, then
  inline completion, then falls back normally.

Snippet behavior:

- LuaSnip loads both VS Code snippets and local snippet files.
- autosnippets are enabled.
- choice nodes show a visual marker.

Local snippet packs currently ship for:

- Go
- Lua

Included examples cover Go error checks, handler templates, structs,

HEALTH, DEVELOPMENT, TROUBLESHOOTING        *onebeer-health*
*onebeer-development*
*onebeer-troubleshooting*

Primary health entrypoint:

>
:checkhealth onebeer
<

The health check gives you a practical readiness sweep across:

- Neovim version requirement
- core external tools such as `git`, `rg`, `brew`, `node`, `go`, `fzf`, `jq`
- JSON-generator dependencies such as `gojson` and `quicktype`
- formatter and linter chains such as `stylua`, `shfmt` / `beautysh`,
  `prettierd` / `prettier`, `selene`, `oxlint`, `shellcheck`, `yamllint`,
  `hadolint`, `gitlint`, `actionlint`, and optional `eslint_d`
- interactive installation offers for supported external and Mason-managed
  tools

Dependency automation notes:

- Homebrew-backed installs are offered for `git`, `rg`, `fzf`, `jq`, and
  `mise`.
- `mise` can be used to install Node.js and Go.
- `go install` is used for `gojson`.
- `npm -g` is used for `quicktype`.
- Mason installs are attempted for selected formatters and linters.

Recommended repo validation commands when you are changing the config:

>
selene .
stylua --check .
nvim --headless "+Lazy! check" +qa
nvim --headless "+checkhealth onebeer" +qa
nvim --headless "+helptags doc" +qa
<

Troubleshooting checklist to work through in order:

- run |OneBeerDoctor|
- open the LSP log with |InspectLog|
- verify `gh auth status` before GitHub workflows
- verify external tools listed by `:checkhealth onebeer`
- rebuild `doc/tags` after editing this helpfile
- if Lua module state looks stale, run |LoaderResetCache|

Linting and formatting behavior:

- Conform owns format-on-save.
- Lua formatting uses `stylua`.
- Python formatting uses `ruff format`.
- Ruby formatting uses `rubocop`.
- Rust formatting uses `rustfmt`.
- Zig formatting uses `zig fmt`.
- SQL formatting uses `sqlfluff`.
- shell formatting prefers `shfmt` with `beautysh` fallback.
- JS/TS/web formatting prefers `prettierd` with `prettier` fallback.
- JS/TS/Astro linting uses a fast InsertLeave path (`oxlint`, optional
  `eslint_d`) and a save-time `eslint` pass.
- Markdown/prose linting runs `markdownlint`, `write-good`, and `woke`.
- Python/Rust/Ruby/Zig diagnostics stay LSP-owned.
- Gleam and SQL intentionally skip `nvim-lint` in wave 1.

LOCAL OVERRIDES AND NEOVIDE                 *onebeer-local*
*onebeer-neovide*

Local overrides are the right home for machine-specific tweaks:

- machine-specific behavior belongs in `lua/onebeer/local.lua`
- that file is loaded last if it exists
- put personal fonts, transparency tweaks, host-specific mappings, or other
  non-shared settings there

Neovide behavior:

- if `vim.g.neovide` is set, OneBeer still calls the normal defaults first
- then it enables:
  - `neovide_transparency = 0.8`
  - `neovide_cursor_trail_length = 0.1`
  - `neovide_remember_window_size = true`

Local guidance for keeping shared config tidy:

- keep shared behavior in tracked files
- keep machine-specific behavior in `onebeer.local`
- use the native helpfile for stable reference and discovery
